# Twintree Design

Twintree is a local-to-remote execution system for local development and
remote build/test. Its core primitive is a btrfs snapshot, not file-level sync.

The local machine owns the live development environment. The remote machine
owns materialized snapshots that can be used to run commands in a bwrap
namespace.

## Goals

- Keep local editing and AI file tools on the local live rootfs.
- Provide explicit CLI commands that humans or AI agents can call to run bash
  on a remote machine.
- Use btrfs snapshots and incremental `btrfs send/receive` as the only sync
  mechanism.
- Make synchronization explicit. Twintree does not infer dirty state and does
  not auto-push before every bash command.
- Keep state visible and minimal: project config, host config, and named
  snapshot checkpoints.
- Implement the CLI and local/remote control path in Rust.

## Non-Goals

- No rsync.
- No file-level manifest sync.
- No hidden dirty tracking.
- No automatic two-way merge.
- No git-like object database, commits, rebase, or history graph.
- No OpenCode plugin or tool-call integration.
- No attempt to intercept or route editor/file operations such as
  read/write/edit/apply_patch.
- No automatic artifact catalog.

## Mental Model

Local:

```text
@rootfs (writable)
  The authoritative live working environment. Humans, editors, and AI file
  tools all read and write this environment directly.

@tt-<project-id>-<branch>-local (read-only)
  The current local checkpoint for the project branch. It is a readonly
  snapshot of the complete local rootfs, not only the project directory. It is
  generated only from @rootfs and is used as the local parent for the next
  incremental push. After a successful push, Twintree reports this checkpoint
  path and updates it to the newly pushed readonly snapshot. It is not the
  active workspace.

@tt-<project-id>-<branch>-<hash-id> (read-only)
  A pulled result checkpoint from the remote workspace. There may be many result
  checkpoints for the same project branch, each identified by a hash id. They
  are readonly local snapshots used for review, diff, and manual merge back into
  @rootfs.
```

Remote:

```text
@tt-<project-id>-<branch>-remote (read-only)
  The current remote checkpoint for the project branch. After a successful
  push, it corresponds to @tt-<project-id>-<branch>-local and acts as the
  readonly parent of the remote workspace. It is a materialized copy of the
  pushed local rootfs snapshot.

@tt-<project-id>-<branch>-workspace (writable)
  The writable remote workspace checkpoint used for remote command execution.
  It is created from @tt-<project-id>-<branch>-remote. When the remote
  checkpoint is updated by push, the old workspace is discarded and a new
  workspace is created from the latest remote checkpoint.
```

Normal flow:

```text
local live rootfs
  --twintree push-->
create a new readonly local checkpoint
incrementally send it to the remote checkpoint
replace the previous local checkpoint with the new one
recreate the remote workspace checkpoint
  --twintree exec-->
execute the command or script in the remote workspace through bwrap
```

Return flow:

```text
remote workspace checkpoint
  --twintree pull-->
create a readonly local result checkpoint
  --manual diff/merge-->
update local live rootfs
```

## Project Configuration

Each project has a `.twintree` file at its project root. The file is the project
root marker, similar to `.git`, and stores only project identity and branch
configuration.

Format: TOML.

Example:

```toml
version = 1
project_id = "p7f3a91c2d8"
current_branch = "main"

[branches.main]
host = "buildbox"

[branches.dev]
host = "buildbox"
```

Fields:

- `version`: config schema version.
- `project_id`: stable project identifier. Generated by `twintree init` from
  real project path, local hostname, current time, and randomness.
- `current_branch`: default branch used by commands when no branch is
  specified.
- `branches`: registered Twintree branches for this project.
- `branches.<name>.host`: host alias used by that branch.

The project id must not be derived from path alone. Two copied projects should
not accidentally share remote state.

Twintree project configuration identifies which project branch and remote host
to use, but it does not limit snapshot contents to the project directory.
Checkpoints are complete rootfs snapshots. The discovered project root is used
for configuration lookup and cwd mapping inside the remote rootfs snapshot.

Validation rules:

- `current_branch` must exist in `branches`.
- Branch names must match `[A-Za-z0-9._-]+`.
- `project_id` must match `[A-Za-z0-9._-]+`.
- Every branch host must resolve through the global host configuration.

Branch configuration should stay minimal. Derived paths such as local
checkpoint names and remote checkpoint names are not stored in `.twintree`; they
are computed from `project_id`, branch name, and the naming convention.

### Project Root Discovery

Commands that operate on a project discover the project root by walking upward
from the current working directory until a `.twintree` file is found. The first
matching file is the project root marker, similar to `.git`.

Rules:

- `twintree init` operates on the current directory and fails if the current
  directory or any ancestor already contains a `.twintree` file.
- All other project commands fail clearly if no `.twintree` file is found in
  the current directory or one of its ancestors.
- The command's project-relative cwd is computed by stripping the discovered
  project root prefix from the process cwd.
- Remote execution reconstructs cwd under the remote workspace as:

  ```text
  /run/twintree/snapshots/@tt-<project-id>-<branch>-workspace/<relative-cwd>
  ```

- If the process cwd is the project root, the remote cwd is the workspace root.
- Symlinks should be resolved with canonical paths before computing the
  relative cwd. If the canonical cwd is not under the canonical project root,
  the command fails.

## Global Configuration

Local rootfs settings and host aliases are configured with a small
git-config-like CLI subset.

Storage path:

```text
~/.twintree/config
```

Example:

```ini
[local]
    rootfs = /run/twintree/rootfs

[host "buildbox"]
    hostname = 10.0.0.12
    user = builder
    port = 22
    key = ~/.ssh/id_ed25519
    sshArgs = -o StrictHostKeyChecking=accept-new
    bwrapFlags = --unshare-pid
```

CLI examples:

```bash
twintree config local.rootfs /run/twintree/rootfs --apply
twintree config host.buildbox.hostname 10.0.0.12 --apply
twintree config host.buildbox.user builder --apply
twintree config host.buildbox.key ~/.ssh/id_ed25519 --apply
twintree config host.buildbox.bwrapFlags --unshare-pid --apply
twintree config host.buildbox.bwrapFlags "" --apply
twintree config host.buildbox.hostname
twintree config --list
twintree config --unset host.buildbox.key --apply
```

There is no separate `twintree host add` command in the first design. Setting
any `host.<alias>.*` key creates that host alias if it does not already exist.

Host configuration only describes how to connect to the remote machine. It does
not define the remote snapshot root. Twintree uses a fixed storage root on both
local and remote machines:

```text
/run/twintree
```

This mirrors the Autopilot style of deploying btrfs subvolumes under a known
runtime root. If a command is invoked from a cwd inside the project:

```text
<project-root>/<relative-cwd>
```

then remote execution uses the corresponding path inside the remote workspace:

```text
/run/twintree/snapshots/@tt-<project-id>-<branch>-workspace/<relative-cwd>
```

The first implementation only needs global host config. Project-local config is
limited to `.twintree`.

`local.rootfs` defines the local live rootfs subvolume that Twintree snapshots
for push. The system must already be installed with btrfs as the root
filesystem, and the live rootfs must be inside a btrfs subvolume. Twintree does
not discover or create this system layout. If `local.rootfs` is missing or does
not point at a btrfs subvolume, commands that need rootfs snapshots fail.

## Snapshot Naming

Twintree uses role-based names for the current local/remote pair, and
hash-based names for imported result checkpoints.

Current local checkpoint:

```text
@tt-<project-id>-<branch>-local
```

Temporary local checkpoint during push:

```text
@tt-<project-id>-<branch>-local.next
```

Remote base checkpoint:

```text
@tt-<project-id>-<branch>-remote
```

Remote writable workspace checkpoint:

```text
@tt-<project-id>-<branch>-workspace
```

Imported local result checkpoint:

```text
@tt-<project-id>-<branch>-<snapshot-id>
```

The `snapshot-id` is generated when pulling a remote workspace/result back to
the local machine. It should be short, unique, and stable enough to reference in
later diff/drop commands.

The exact btrfs subvolume mount path can be configured later. The first
implementation assumes the configured local user and remote host user have the
permissions required to create, send, receive, snapshot, and delete the needed
btrfs subvolumes.

## Remote Base and Workspace Snapshots

`btrfs receive` does not overwrite an existing subvolume. It materializes the
sent stream as a new subvolume under the receive directory. For Twintree, this
means remote updates should use a staging directory and only promote the
received snapshot after the receive command succeeds.

Remote storage layout:

```text
/run/twintree/
  incoming/
  locks/
    <project-id>-<branch>.lock
  snapshots/
    @tt-<project-id>-<branch>-remote
    @tt-<project-id>-<branch>-workspace
```

`incoming/` is a normal staging directory. Received subvolumes are promoted into
`snapshots/` after a successful receive. The first implementation should keep
all Twintree subvolumes under the same btrfs filesystem so promotion can be a
same-filesystem `mv`.

### Concurrency and Locks

Twintree uses one remote advisory lock per project branch:

```text
/run/twintree/locks/<project-id>-<branch>.lock
```

The lock is acquired with `flock` on the remote host. The first implementation
may invoke remote helper commands through `flock` rather than running a daemon.

Rules:

- `twintree push`, `twintree pull`, `twintree exec`, and `twintree branch delete`
  must acquire the branch lock before reading or mutating remote branch state.
- `twintree exec` holds the lock for the full lifetime of the remote command.
  This prevents push or branch delete from replacing or deleting the workspace
  while a command is running.
- `twintree push` holds the lock from preflight cleanup through receive,
  promotion, workspace recreation, and remote success reporting.
- `twintree pull` holds the lock while snapshotting/sending the remote
  workspace, so it cannot race with push or branch delete.
- A second concurrent operation for the same project branch fails fast with a
  clear "branch is locked" error unless a later CLI option explicitly requests
  waiting.
- Locks are per project branch. Different branches may run concurrently.

### Receive Behavior

First push:

```text
remote incoming dir is created
btrfs receive /run/twintree/incoming
received snapshot appears as:
  /run/twintree/incoming/@tt-<project-id>-<branch>-local
promote received snapshot to:
  /run/twintree/snapshots/@tt-<project-id>-<branch>-remote
create workspace snapshot from remote checkpoint
```

Subsequent push:

```text
btrfs receive /run/twintree/incoming
received snapshot appears as:
  /run/twintree/incoming/@tt-<project-id>-<branch>-local.next
validate receive success
delete old workspace checkpoint
delete old remote checkpoint
promote received snapshot to:
  /run/twintree/snapshots/@tt-<project-id>-<branch>-remote
create new workspace checkpoint from remote checkpoint
clear incoming
```

If the remote checkpoint already exists, receive must not target a path that
would collide with it. Use `incoming` to avoid partial updates and name
conflicts.

### Failed Push Recovery

Push must be transactional at the checkpoint level: the current local checkpoint
and the current remote checkpoint are advanced only after the new remote
checkpoint and workspace have been promoted successfully.

Staging names are branch-specific:

```text
/run/twintree/incoming/@tt-<project-id>-<branch>-local
/run/twintree/incoming/@tt-<project-id>-<branch>-local.next
```

Recovery rules:

- At the start of `twintree push`, after acquiring the branch lock and before
  `btrfs receive`, remove any stale incoming subvolume or directory whose name
  matches the expected receive name for that branch.
- If `btrfs receive` fails or SSH disconnects before remote promotion completes,
  the command fails and does not rename local `.next` to the current local
  checkpoint.
- On push failure, Twintree should attempt best-effort cleanup of the local
  `.next` checkpoint and the remote incoming receive path. Cleanup failure is
  reported as a warning, not as success.
- The old local checkpoint is preserved until remote promotion and workspace
  recreation both succeed. It remains the incremental parent for the next push.
- The old remote checkpoint and workspace are preserved until the new received
  snapshot has been validated. If promotion or workspace recreation fails after
  deleting old remote state, the command reports that the branch requires manual
  recovery; the first implementation should avoid this by validating all paths
  and deleting/replacing only while holding the branch lock.
- A later `twintree push --full` may recover from parent mismatch, but Twintree
  does not automatically switch from incremental to full send.

### Remote Checkpoint

`@tt-<project-id>-<branch>-remote` is the latest successfully pushed readonly
base checkpoint for a project branch.

Rules:

- The remote checkpoint is aligned with the current local checkpoint after a
  successful push.
- The remote checkpoint is replaced only by `twintree push`.
- The remote checkpoint should be treated as immutable after promotion.
- A push deletes and recreates the workspace checkpoint after replacing the
  remote checkpoint.

### Workspace Checkpoint

`@tt-<project-id>-<branch>-workspace` is a writable child snapshot of the
remote checkpoint.

After every successful push:

```text
delete old workspace checkpoint if it exists
btrfs subvolume snapshot \
  @tt-<project-id>-<branch>-remote \
  @tt-<project-id>-<branch>-workspace
```

Remote execution uses the workspace checkpoint:

```text
bwrap --bind @tt-<project-id>-<branch>-workspace / ...
```

Only one workspace checkpoint is kept for each project branch. It is preserved
across multiple `twintree exec` commands and discarded/recreated when the base
remote checkpoint is updated.

### Imported Result Checkpoints

Pulling the remote workspace back to the local machine creates a new readonly
result checkpoint:

```text
@tt-<project-id>-<branch>-<snapshot-id>
```

There may be many imported result checkpoints. This is useful for comparing
multiple remote build/test outcomes, but it can consume disk space because old
snapshots keep referenced extents alive. GC should be explicit in the first
version.

Initial GC policy:

- Keep all imported result checkpoints until explicit deletion.
- Provide `twintree list` so users can see local result checkpoints and their
  approximate disk usage when requested.
- Provide `twintree du` so users can sort checkpoints by estimated exclusive
  and referenced size.
- Provide `twintree drop <snapshot-id>` for explicit deletion.
- Add retention policy later, such as keep latest N per branch.

## Commands

### Dry-Run and Apply

Commands that change Twintree configuration or snapshots are dry-run by default.
They print the actions they would take and exit without applying changes unless
`--apply` is provided.

Mutating commands:

- `twintree config <key> <value> --apply`
- `twintree config --unset <key> --apply`
- `twintree branch create <name> --host <host-alias> --apply`
- `twintree branch switch <name> --apply`
- `twintree branch set-host <name> <host-alias> --apply`
- `twintree branch delete <name> --apply`
- `twintree push --apply`
- `twintree pull --apply`
- `twintree drop <snapshot-id> --apply`

`twintree init` is not dry-run by default. It creates `.twintree` and the
initial `main` branch config directly. The command is expected to be run once at
project setup time.

Read-only commands such as `twintree status`, `twintree list`, `twintree du`,
and `twintree branch` do not need `--apply`.

`twintree exec` is not dry-run by default because its primary purpose is to run
the user's command in the remote workspace. It still does not modify local
Twintree state.

Dry-run output should include:

```text
project id
branch
host alias
local paths that would be created/deleted/renamed
remote paths that would be created/deleted/renamed
whether a remote lock would be acquired
whether the planned operation is full or incremental
```

Dry-run must not create durable snapshots, delete snapshots, update `.twintree`,
write host config, receive btrfs streams, or run remote bwrap commands.

### `twintree config`

Configure host aliases in `~/.twintree/config`.

```bash
twintree config <key>
twintree config <key> <value> --apply
twintree config --list
twintree config --unset <key> --apply
```

The first version intentionally implements only a small git-config-like subset.
It always reads and writes `~/.twintree/config`; there is no `--global`,
`--local`, or `--system` scope.

Supported key pattern:

```text
local.<field>
host.<alias>.<field>
```

Supported local keys:

- `rootfs`

Supported host keys:

- `hostname`
- `user`
- `port`
- `key`
- `sshArgs`
- `bwrapFlags`

Setting any `host.<alias>.*` key creates that host alias if it does not already
exist.

`host.<alias>.bwrapFlags` configures additional bwrap isolation flags for
`twintree exec` on that remote host:

- if the key is absent, Twintree uses the default value `--unshare-pid`;
- if the key exists with a non-empty value, Twintree parses the value into
  shell-like words and appends those flags without applying an allowlist;
- if the key exists with an empty value, Twintree appends no extra isolation
  flags, which explicitly disables the default PID namespace isolation.

`bwrapFlags` is a free-form host policy override. Invalid, contradictory, or
unsafe bwrap flags are user error and their behavior is undefined by Twintree.
Twintree's responsibility is to pass the configured tokens to bwrap faithfully
and fail closed if bwrap itself fails.

The git-config-like storage format supports an empty override as either an
empty value in the config file or a CLI set operation with an empty string:

```bash
twintree config host.buildbox.bwrapFlags "" --apply
```

`twintree config --list` should print one key/value pair per line:

```text
host.buildbox.hostname=10.0.0.12
host.buildbox.user=builder
host.buildbox.key=~/.ssh/id_ed25519
host.buildbox.bwrapFlags=--unshare-pid
```

### `twintree init <host-alias>`

Initialize the current directory as a Twintree project.

Actions:

1. Resolve the project root.
2. Generate a unique project id.
3. Write `.twintree`.
4. Create `branches.main` with the given host alias.
5. Set `current_branch` to `main`.
6. Validate that `local.rootfs` and the host alias are configured.

`twintree init <host-alias>` is equivalent to initializing project identity and
creating the `main` branch with `twintree branch create main --host
<host-alias>`.

Output should include:

```text
project id
host alias
current branch
local checkpoint name:
  @tt-<project-id>-<branch>-local
```

### `twintree branch`

Show the current branch.

### `twintree branch create <name> --host <host-alias> [--apply]`

Create a new project branch in `.twintree`.

This does not create a snapshot immediately. The branch checkpoint is created on
the first `twintree push`.

Branch creation validates that `local.rootfs` and the host alias are configured
before updating `.twintree`. It may also check baseline remote capabilities:

```text
ssh connectivity as configured host user
bash
btrfs
bwrap
flock
fixed storage root: /run/twintree
permission to create incoming/, locks/, and snapshots/
```

If configuration validation fails, branch creation fails before modifying
`.twintree`.

### `twintree branch switch <name> [--apply]`

Switch `current_branch` in `.twintree`.

The branch must already exist in the `branches` table.

### `twintree branch set-host <name> <host-alias> [--apply]`

Update the host alias for an existing branch.

### `twintree branch delete <name> [--apply]`

Delete a project branch and its current local/remote checkpoints.

Preflight:

1. Resolve the branch host.
2. Fail if the branch is the current branch.
3. Connect to the remote host.
4. Acquire the remote branch lock.
5. Validate that the remote Twintree storage root is reachable.

If any preflight step fails, `branch delete` must not modify `.twintree` or
delete local checkpoints.

Actions after successful preflight:

1. Delete the remote workspace checkpoint if it exists:
   `@tt-<project-id>-<name>-workspace`
2. Delete the remote checkpoint if it exists:
   `@tt-<project-id>-<name>-remote`
3. Delete the local checkpoint if it exists:
   `@tt-<project-id>-<name>-local`
4. Remove `branches.<name>` from `.twintree`.
5. Print warnings for any remaining local result checkpoints:
   `@tt-<project-id>-<name>-<snapshot-id>`

Failure contract:

- If remote deletion fails, stop before deleting local checkpoints or modifying
  `.twintree`.
- If local checkpoint deletion fails after remote deletion succeeded, stop
  before modifying `.twintree` and report the branch as partially deleted:
  remote state is gone, local config remains.
- If updating `.twintree` fails after checkpoint deletion succeeded, report the
  branch as partially deleted and print the exact remaining config path to fix.
- Successful completion means remote current checkpoints, local current
  checkpoint, and branch config have all been removed.

Deleting a branch does not delete local result checkpoints. They become
dangling review snapshots and must be dropped explicitly:

```bash
twintree drop <snapshot-id> --branch <name> --apply
```

If the branch is the current branch, the command should fail until the user
switches to another branch.

If the remote host cannot be reached, the first version should fail before
modifying `.twintree`.

There is no `branch delete --all` in the first design.

### `twintree push [--branch <branch>] [--full] [--apply]`

Push the local live rootfs snapshot to the remote branch checkpoint. This is a
complete rootfs snapshot from `local.rootfs`, not a project-directory-only
snapshot.

If no branch is given, use `.twintree.current_branch`.

Strategy:

- If the local checkpoint or remote checkpoint is missing, push uses a full
  `btrfs send`.
- If both checkpoints exist, push uses incremental `btrfs send -p`.
- `--full` forces a full `btrfs send` even if both checkpoints exist.
- If incremental receive fails, the command fails clearly. It does not
  automatically retry as full; the user can run `twintree push --full`.

Preflight diagnostics:

- Before an incremental push, verify that the local parent checkpoint exists and
  is readonly.
- Before an incremental push, verify that the remote checkpoint exists and is
  readonly.
- If either parent is missing, print which side is missing and use a full push
  unless the user explicitly requested an incremental-only mode in a later CLI.
- If `--full` is not provided and both parents exist, print the local parent and
  remote parent paths that will be used for the incremental send.

`twintree init` and `twintree branch create --apply` do not create branch
snapshots. The first push for a branch is therefore a full push.

First push:

```text
create readonly checkpoint from local.rootfs:
  @tt-<project-id>-<branch>-local

btrfs send checkpoint | ssh host btrfs receive incoming/
remote promote received incoming snapshot to:
  @tt-<project-id>-<branch>-remote
remote create workspace checkpoint:
  @tt-<project-id>-<branch>-workspace
```

Subsequent push:

```text
old = @tt-<project-id>-<branch>-local
new = @tt-<project-id>-<branch>-local.next

create readonly snapshot from local.rootfs -> new
btrfs send -p old new | ssh host btrfs receive incoming/
remote delete old workspace checkpoint
remote delete old remote checkpoint
remote promote received incoming snapshot to:
  @tt-<project-id>-<branch>-remote
remote create new workspace checkpoint:
  @tt-<project-id>-<branch>-workspace
delete old
rename new -> old
print old checkpoint path
```

Important ordering:

```text
Do not delete the old checkpoint before send completes.
The old checkpoint is the incremental parent.
```

If incremental send or receive fails, fail with diagnostics. The error output
should include:

- whether the failure happened during local `btrfs send`, SSH transport, remote
  `btrfs receive`, promotion, or workspace recreation;
- the local parent path used for `btrfs send -p`;
- the expected remote parent path;
- the remote incoming path used for receive;
- a short stderr excerpt from `btrfs send` or `btrfs receive`, if available;
- whether Twintree cleaned up the local `.next` checkpoint and remote incoming
  path;
- the next suggested command, usually `twintree push --full` for parent
  mismatch.

Twintree does not try to prove whether the mismatch was caused by manual remote
snapshot edits, deleted local parents, copied project ids, or an interrupted
older command. It reports the observed missing or rejected parent state and
leaves recovery explicit.

### `twintree exec [--branch <branch>] [--interactive | --file <path> | --script <text> | -- <command>]`

Execute a command, script, or interactive shell in the remote workspace
checkpoint for the branch.

Forms:

```bash
twintree exec -- make test
twintree exec --file ./scripts/check.sh
twintree exec --script 'make test && cargo test'
twintree exec --interactive
```

Remote actions:

```text
use existing workspace checkpoint:
  @tt-<project-id>-<branch>-workspace
bwrap bind workspace checkpoint as /
mount /dev, /proc, /sys, and /tmp for the namespace
execute command, script, or interactive shell with bash
return stdout, stderr, and exit code
```

#### bwrap Sandbox Specification

Twintree wraps every `exec` command in a bwrap namespace on the remote host. The
sandbox is constructed as follows:

```text
bwrap \
  --bind <workspace-checkpoint> / \
  --dev /dev \
  --proc /proc \
  --ro-bind /sys /sys \
  --tmpfs /tmp \
  <bwrap-flags> \
  --chdir <workspace-relative-cwd> \
  bash ...
```

`<bwrap-flags>` comes from the selected host's `bwrapFlags` config:

- absent config key: `--unshare-pid`;
- non-empty config value: parse the configured flags as shell-like words and
  append them without applying an allowlist;
- empty config value: append no extra flags.

The configured flags are host-level policy because the flags must match the
remote kernel, distro, and build/test requirements. Project config does not
override bwrap isolation in the first design.

Invalid, contradictory, or unsafe configured flags are user error and have
undefined behavior from Twintree's perspective. Twintree fails closed if bwrap
returns an error.

Twintree should pass parsed flag tokens as `Command` arguments, not concatenate
them into a shell command. The only intentional shell boundary remains the user
command passed to `bash -lc` in `twintree exec -- <command>`.

Mode rules:

- `-- <command>` passes the command string intentionally to `bash -lc`.
- `--file <path>` reads a local script file and sends its content to the remote
  side for execution inside the workspace. The file path is resolved from the
  local cwd before execution.
- `--script <text>` executes the provided script text inside the workspace.
- `--interactive` allocates a TTY and starts an interactive shell inside the
  workspace at the mapped project-relative cwd.
- Exactly one execution mode is allowed.
- `--interactive` returns the remote shell exit code and streams stdin, stdout,
  and stderr directly.

The command or shell may modify the remote workspace checkpoint. It does not
update local state.

There is no `twintree run` alias in the first design.

### `twintree pull [--branch <branch>] [snapshot-id] [--apply]`

Pull the remote workspace checkpoint back to the local machine as an imported
readonly result checkpoint for review.

Example local name:

```text
@tt-<project-id>-<branch>-<snapshot-id>
```

This command does not merge anything into the live rootfs.

If no `snapshot-id` is provided, Twintree generates one and prints it.

### `twintree status [--branch <branch>] [--remote]`

Show the local project state relative to the current local checkpoint.

If `--branch` is not provided, use `.twintree.current_branch`.

The first version must not maintain hidden dirty tracking. `status` answers the
question by comparing the live project tree with the current local checkpoint at
command time.

Output should include:

```text
project id
branch
project root
current local checkpoint path, or "missing"
local changes since checkpoint: yes/no/unknown
remote checkpoint path and workspace path, if --remote is provided
pending local .next checkpoint, if one exists
stale remote incoming path, if --remote is provided and one exists
suggested next command
```

Local change detection should use btrfs-aware comparison when practical, such
as `btrfs subvolume find-new` or a readonly temporary snapshot plus `btrfs
send --no-data`/metadata comparison. If the filesystem cannot provide a reliable
answer, print `unknown` with the reason rather than silently reporting clean.

`status` does not push, pull, create durable checkpoints, delete stale state, or
modify `.twintree`.

### `twintree list [--branch <branch>] [--local] [--remote] [--sizes]`

List local and optionally remote Twintree checkpoints.

The first version should show at least:

```text
current local checkpoint
imported local result checkpoints
remote checkpoint
remote workspace checkpoint
```

With `--sizes`, include approximate referenced and exclusive byte counts when
available. Btrfs snapshots share extents, so size output must label whether a
number is:

- `referenced`: data reachable from the checkpoint, including shared extents;
- `exclusive`: data that would likely be freed by deleting only that checkpoint;
- `unknown`: size could not be computed cheaply or safely.

`list --sizes` is for visibility, not cleanup. It must not delete snapshots.

### `twintree du [--branch <branch>] [--local] [--remote]`

Show disk usage for Twintree checkpoints, sorted by estimated exclusive size
descending when available.

The first version should include:

```text
checkpoint role
checkpoint id/name
location: local or remote
referenced bytes
exclusive bytes
path
```

Implementation may use `btrfs qgroup show`, `btrfs filesystem du`, or another
btrfs-native size query. If quotas/qgroups are required for exclusive size and
are not enabled, print `unknown` for exclusive size and include a short reason.

`du` does not imply retention policy. Users still reclaim space explicitly with
`twintree drop <snapshot-id>` or branch deletion.

### `twintree drop <snapshot-id> [--branch <branch>] [--apply]`

Delete a local result checkpoint:

```text
@tt-<project-id>-<branch>-<snapshot-id>
```

If `--branch` is not provided, use `.twintree.current_branch`.

`drop` does not delete local checkpoints, remote checkpoints, remote workspace
checkpoints, or branch configuration.

## AI Usage Model

Twintree is a standalone CLI. It does not integrate with OpenCode or intercept
tool calls.

Humans and AI agents use the same explicit commands:

```bash
twintree status
twintree push --apply
twintree exec -- make test
twintree exec --file ./scripts/check.sh
twintree exec --interactive
twintree pull failed-test --apply
```

Editor operations and AI file operations happen directly on the local live
rootfs. Remote execution only happens when the caller explicitly invokes
`twintree exec`.

The caller is responsible for pushing before remote execution when local
changes should be visible remotely:

```bash
twintree status
twintree push --apply
twintree exec -- make test
```

## Safety Rules

- Fail closed if remote bwrap setup fails. Do not silently execute on the local
  host.
- Print the local checkpoint path after every successful push.
- Print the remote checkpoint and workspace checkpoint paths after every
  successful push.
- Refuse to replace a remote checkpoint while a command is running in the
  workspace checkpoint.
- Require explicit deletion for snapshots.
- Do not automatically pull or merge remote changes into the local live rootfs.
- Validate project ids, branch names, and host aliases with strict character
  allowlists.

Suggested name allowlist:

```text
[A-Za-z0-9._-]+
```

## Rust Implementation Plan

The CLI should be implemented in Rust.

Suggested crates:

- `clap` for CLI parsing.
- `serde` and `toml` for `.twintree`.
- `dirs` or `home` for locating `~/.twintree`.
- `anyhow` or `eyre` for early error handling.
- `thiserror` if/when errors become structured.
- `rand` plus `sha2` or `blake3` for project id generation.

Suggested module layout:

```text
src/
  main.rs        CLI entrypoint
  cli.rs         clap command definitions
  config.rs      global config and project config
  project.rs     project discovery, id generation, branch handling
  host.rs        host alias resolution and ssh command construction
  btrfs.rs       local btrfs operations
  remote.rs      ssh remote operations
  exec.rs        remote bwrap execution
```

Prefer `std::process::Command` with argument arrays. Avoid shell string
construction except for the final user command passed intentionally to
`bash -lc`.
